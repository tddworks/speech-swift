# Hibiki Zero-3B Speech Translation Inference Pipeline

## Overview

```
Source audio (24kHz) → Mimi encoder (12.5Hz, 16 codebooks)
                  ↓
           33-stream temporal transformer (GQA, 28 layers, dim=2048)
                  ↓
           Scheduled depformer (6 layers, 16 codebook steps)
                  ↓
           Mimi decoder (12.5Hz, 16 codebooks) → English audio (24kHz)
```

Hibiki Zero-3B is a streaming speech-to-speech translation model (FR/ES/PT/DE → EN) built on the Kyutai Moshi/Mimi stack. The temporal transformer consumes the previously sampled text token and target audio codes as autoregressive input at each step, alongside the streaming source audio codes. Generation terminates when the model samples text-EOS past the source-window.

See [docs/models/hibiki.md](../models/hibiki.md) for full architecture details.

## Pipeline Stages

### 1. Mimi Encode

The source PCM (24 kHz mono) is fed through the streaming Mimi codec to produce a `(1, 16, T)` tensor of RVQ codes at 12.5 Hz. Each 80 ms input frame yields one timestep of 16 codebook indices.

### 2. Token Cache Prefill

Hibiki maintains a 33-stream token cache (text + 16 target audio + 16 source audio). The 16 source codebooks are prefilled into streams 17–32, offset by their per-stream delay (`[0, 2, 2, ..., 2]` — cb0 delay 0, cb1–15 delay 2). The cache is initialized with each stream's init token (`textInitialTokenId = textCard` for text; `initialTokenId = card` for audio), mirroring upstream Moshi's `state.initial`.

### 3. Temporal Transformer Step

At each step the transformer reads one token from each of the 33 streams at index `step` (with init-token substitution when `step <= delays[k]`, mirroring `lm.py:698-702`). The transformer's `forward` produces a hidden state and text logits. Text token is sampled (greedy if `HIBIKI_GREEDY=1`, otherwise temperature/top-k from `cfg.sampling`).

The sampled text token and the 16 target codes generated by the depformer in step (3b) are written into the cache at **index `step + 1`** — the next iteration's read slot. This is the autoregressive feedback path: Hibiki at step `t+1` sees the token it sampled at step `t`.

#### 3b. Depformer Step

For each step the depformer runs 16 sub-steps, one per target codebook. The scheduled-multi-linear projection uses a 9-slice weight bank with schedule `[0..7, 8×8]`. The first sub-step takes the temporal-transformer hidden + text embedding as input; subsequent sub-steps take the hidden + the previous-codebook embedding.

### 4. EOS Detection

Hibiki's training emits SPM padding (id 3) while it accumulates source context, then emits content tokens, then a text-EOS (id 2). The decode loop terminates the moment EOS is sampled AND `step > tSrc`. Early EOS sampled during the audio-streaming window is harmless because the weight loader aliases `text_emb[2]` (EOS) onto `text_emb[3]` (PAD) — mirroring upstream `loaders.py:312`. A safety cap at `max(tSrc * 5/2, tSrc + 20)` steps guards against runaway generation.

### 5. Un-shift + Mimi Decode

The generated target codebooks are un-shifted by their per-stream delay (cb0 at `t`, cb1–15 at `t+2`) and fed to the Mimi decoder, producing 24 kHz mono English audio.

## Language Support

| Source | Status | Test fixture |
|---|---|---|
| FR | strict canary | FLEURS-style `fleurs_fr.wav` (3.5s) |
| ES | strict canary | Hibiki official sample (europarl_st/5dc1d533, 24 kHz TTS, 5s trim) |
| PT | warn-only (content-faithful but lower keyword recall) | FLEURS-style `fleurs_pt.wav` (5.2s) |
| DE | warn-only (content-faithful but lower keyword recall) | FLEURS-style `fleurs_de.wav` (5.4s) |

**FLEURS Spanish is out-of-distribution.** Both Python upstream and the Swift port produce degenerate output on FLEURS-ES (Python emits 1643 generation steps / ~131 s of broken audio without sampling EOS). The ES test fixture is therefore a 5 s excerpt from Kyutai's official samples space which matches the training distribution (24 kHz TTS-generated).

## CLI

```bash
# Translate one file (greedy + verbose for reproducibility)
HIBIKI_GREEDY=1 speech audio-translate input_fr.wav -o out_en.wav --source-lang fr --verbose

# 8-bit quantization (lower memory, slightly higher quality vs default 4-bit)
speech audio-translate input.wav --quantization 8bit -o out.wav --source-lang es

# Print inner-monologue raw token IDs (SPM decoder is a follow-up)
speech audio-translate input.wav -o out.wav --transcript
```

## Environment Variables

| Var | Effect |
|---|---|
| `HIBIKI_GREEDY=1` | Force argmax decoding for both text and target audio. Used by the strict E2E canaries. |
| `HIBIKI_E2E=1` | Enable the E2E test cases (requires ~2.7 GB model download). |
| `HIBIKI_STRICT_ALL=1` | Promote PT/DE tests from warn-only to strict. |
| `HIBIKI_LENIENT=1` | Demote FR/ES tests from strict to warn-only (debugging only). |
| `HIBIKI_MODEL_ID=<repo>` | Override the default `aufklarer/Hibiki-Zero-3B-MLX-4bit` model id. |

## Performance (M2 Max, MLX 4-bit)

| Metric | Greedy | Sampled |
|---|---|---|
| Per-step latency | ~75 ms | ~95 ms |
| Source duration → wall-clock (FR fleurs, 3.54 s) | ~5 s | ~7 s |
| Output duration | ~1.0–1.6× source | ~1.0–1.6× source |
| Generation length cap | `max(tSrc * 5/2, tSrc + 20)` steps | same |

## Known Limitations

- **`translateStream()` returns a single chunk** — the streaming entry point currently wraps the offline `translate()` and emits one final `AudioChunk`. True per-chunk Mimi streaming decode is a v2 follow-up.
- **No SentencePiece text decoder** — `--transcript` prints raw token IDs. SPM decode wiring (using the bundled `tokenizer_spm_48k_multi6_2.model`) is a follow-up.
- **Sampled mode noticeably noisier than greedy** — reproducible runs and CI canaries use `HIBIKI_GREEDY=1`.
